Multi-Chip Tips and Tricks
==========================

Discovered while implementing 2-chip tensor parallelism for Oasis-500M.

FABRIC INITIALIZATION (CRITICAL)
=================================
Must call ttnn.set_fabric_config() BEFORE opening the mesh device:

    ttnn.set_fabric_config(ttnn.FabricConfig.FABRIC_1D)
    mesh_device = ttnn.open_mesh_device(ttnn.MeshShape(1, N_CHIPS))

Without this, all_reduce and all_gather fail with:
    TT_FATAL: this->fabric_context_ != nullptr

Available FabricConfig options:
    DISABLED, FABRIC_1D, FABRIC_1D_NEIGHBOR_EXCHANGE, FABRIC_1D_RING, FABRIC_2D, CUSTOM

For 2-chip linear topology, FABRIC_1D works.

MESH DEVICE SETUP
==================
    mesh_device = ttnn.open_mesh_device(
        ttnn.MeshShape(1, N_CHIPS),
        trace_region_size=100000000,  # if using trace
    )

The mesh device object is used everywhere a single device was used before.
Most ttnn ops work transparently on mesh tensors.

TENSOR DISTRIBUTION
====================
Replicate (same on all chips):
    ttnn.from_torch(t, ..., device=mesh_device,
        mesh_mapper=ttnn.ReplicateTensorToMesh(mesh_device))

Shard along dimension:
    ttnn.from_torch(t, ..., device=mesh_device,
        mesh_mapper=ttnn.ShardTensorToMesh(mesh_device, dim=1))

Readback (concat chips along dim):
    ttnn.to_torch(t, mesh_composer=ttnn.ConcatMeshToTensor(mesh_device, dim=0))

COLLECTIVES
============
all_reduce: sums partial results across chips. Output replicated.
    result = ttnn.all_reduce(partial_tensor)

all_gather: concatenates sharded results. Output replicated.
    result = ttnn.all_gather(sharded_tensor, dim=1)

Both require fabric to be initialized (see above).

MATMUL ON MESH
===============
ttnn.matmul works transparently:
- replicated x col-sharded -> col-sharded output (column parallel)
- col-sharded x row-sharded -> partial sum per chip (row parallel, needs all_reduce)

Verified max_diff ~2-6 for (160, 1024) x (1024, 5120) matmul, comparable to single chip.

TT-LANG ON MESH
=================
TT-Lang kernels work transparently on mesh tensors. Each chip runs the
kernel independently on its local tensors. No kernel code changes needed.

NOTE: TT-Lang compiler does not like `tensor.shape[N]` inside nested
functions in the kernel. Use a captured variable instead:
    WRONG:  n_cols = a.shape[1] // TILE   (inside dm_read)
    RIGHT:  Use the variable from kernel scope.

TRACE ON MESH
==============
Trace capture and execution work on mesh_device:
    trace_id = ttnn.begin_trace_capture(mesh_device, cq_id=0)
    ...
    ttnn.end_trace_capture(mesh_device, trace_id, cq_id=0)
    ttnn.execute_trace(mesh_device, trace_id, cq_id=0, blocking=False)

TTNN OPS ON MESH
=================
Verified working: matmul, add, gelu, slice, concat, reshape, permute.
All operate per-chip on local tensor data.

ZERO-ALLOCATION PATTERN
========================
For the traced path, all tensors must be pre-allocated. With mesh:
- Use mesh_mapper when creating tensors with from_torch
- Use optional_output_tensor for matmul (writes into pre-allocated mesh tensor)
- Use output_tensor for add

CLOSING
========
    ttnn.close_mesh_device(mesh_device)   # NOT ttnn.close_device()

TRACE + COLLECTIVES
====================
Trace with all_reduce and all_gather WORKS on mesh. The earlier test
failure was a bug in the test (wrong shard dimensions), not a runtime issue.

The existing oasis code pre-allocates all tensors before trace capture.
With mesh, ensure all shard/replicate tensors are also pre-allocated.

NUMERICAL PRECISION
====================
Column-parallel matmul (160x1024 @ 1024x5120): max_diff ~2-3 vs fp32
Row-parallel + all_reduce (160x4096 @ 4096x1024): max_diff ~4-6 vs fp32
Chained col+row: max_diff ~98 vs fp32 (BF16 compounds, but real weights
are much smaller magnitude than random normal, so this is fine in practice).

The all_reduce itself is exact (inter-chip diff = 0.0).
The all_gather is exact (no computation, just data movement).

PERFORMANCE NOTES
==================
Results on Oasis-500M DiT (4 DDIM steps, T=2):
- 1 chip + trace: 49ms/step (20.3 FPS)
- 2 chips + trace: 38ms/step (26.0 FPS) -- 22% speedup
- 2 chips, no trace: 80ms/step (dispatch overhead dominates)

Key: trace is essential for multi-chip perf. Without trace, the 660+
dispatches per step (now 2x for collectives) overwhelm any compute savings.

TT-Lang kernels required ZERO code changes for multi-chip. Only the
n_heads_val factory parameter changed (16 -> 16/N_CHIPS).

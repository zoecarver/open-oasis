# Oasis 500M on Tenstorrent

Real-time Minecraft world generation on Tenstorrent Blackhole hardware using [TT-Lang](https://github.com/tenstorrent/tt-lang/tree/main) kernels.

Oasis is a diffusion transformer (DiT) world model that takes keyboard input and generates Minecraft gameplay autoregressively. This repo implements end-to-end inference -- DiT denoising, on-device VAE decode, and video output -- running at **6.3 FPS** (29ms per diffusion step) on a single Blackhole chip.

*See [docs/minecraft_4x.mov](docs/minecraft_4x.mov) for a demo video.*

## Performance

| Metric | Value |
|---|---|
| Diffusion step | 29ms |
| End-to-end (4 DDIM steps + VAE) | 160ms/frame |
| Throughput | 6.3 FPS |
| Hardware | 1x Blackhole chip |

Multi-chip tensor parallelism is supported. Set `N_CHIPS` in `oasis_inference.py` to the number of chips (e.g. `N_CHIPS = 4`). The pipeline automatically shards QKV and MLP weights across chips with all-reduce for the row-parallel projections.

## Setup

### Dependencies

- [tt-lang](https://github.com/tenstorrent/tt-lang/tree/main) (includes `ttl`, `ttnn`, and device drivers)
- PyTorch
- `einops`, `safetensors`, `Pillow`
- `ffmpeg` (optional, only for stitching frames into video)

### Weights

Download the Oasis 500M weights from [Hugging Face](https://huggingface.co/Etched/oasis-500m):

```bash
huggingface-cli download Etched/oasis-500m oasis500m.safetensors
huggingface-cli download Etched/oasis-500m vit-l-20.safetensors
```

The inference script looks for weights in the default HuggingFace cache directory:
```
~/.cache/huggingface/hub/models--Etched--oasis-500m/blobs/
```

You also need the VAE model definition (`vae.py`) from the [open-oasis](https://github.com/etched-ai/open-oasis) repo. Place it where the script can find it (e.g. `/tmp/vae.py` or alongside the inference script).

### Prompt image and actions

Place a prompt image and (optionally) recorded actions in `/tmp`:
- `/tmp/sample_image_0.png` -- 640x360 RGB image (first frame)
- `/tmp/sample_actions_0.one_hot_actions.pt` -- recorded keyboard actions (optional, falls back to zero actions)

## Running

### Batch inference

Generates 30 frames from a prompt image and saves them as PNGs + mp4:

```bash
python oasis_inference.py
```

Output goes to `/tmp/oasis_video_4step_T2/`.

### Interactive play

Keyboard-driven real-time generation:

```bash
python play.py
```

### Play server

HTTP server for browser-based play:

```bash
python play_server.py
```

## Architecture and TT-Lang Kernels

Oasis 500M is a spatio-temporal diffusion transformer (DiT) with a ViT-based VAE. Each video frame is generated by running 4 DDIM denoising steps through the DiT, then decoding the result through the VAE -- all on device inside a single captured trace.

### End-to-end pipeline

```
Prompt image (640x360)
        |
  [VAE encode]  .............. CPU (PyTorch, runs once)
        |
        v
  Latent (18x32x16)
        |
  [patch_embed]  ............. CPU (Conv2d, host->device transfer)
        |
        v
  Context window (T-1 frames of patch embeddings, on device)
        |
  For each new frame:
  |
  |  Noise (random, host->device)
  |     |
  |     v
  |  x4 DDIM steps:  ........ all on device (traced)
  |  |  [patch_embed]           output-space -> input-space via matmul (ttnn)
  |  |  [concat context]        prepend context frames (ttnn)
  |  |  [dit_forward]           16 DiT blocks (see below)
  |  |  [ddim_update]           v-prediction arithmetic (ttnn)
  |  |
  |  v
  |  Denoised patches (160, 64)
  |     |
  |  [bridge_unpatchify] .... on-device layout transform via 3 matmuls (ttnn)
  |     |
  |     v
  |  VAE input (576, 32)
  |     |
  |  [vae_decode]  .......... 12 transformer blocks on device (see below)
  |     |
  |     v
  |  Pixel patches (576, 1200)
  |     |
  |  [unpatchify + clamp] ... CPU readback + reshape
  |     |
  |     v
  |  Frame (360x640x3)
```

### DiT forward pass (per DDIM step)

The DiT has 16 blocks, each with a spatial sub-block and a temporal sub-block. Conditioning (timestep embedding + action embedding) is computed once on the host and reused across all blocks.

```
Conditioning:
  timestep -> freq_embed -> MLP -> SiLU .... host (fp32 for precision)
  action -> linear -> add to cond ........... host
  cond -> SiLU  ............................. TT-Lang (silu_kernel)

Per sub-block (x32: 16 spatial + 16 temporal):
  |
  |-- [adaLN params]                         ttnn.linear + concat
  |-- [LayerNorm + adaLN modulate]           ttnn.layer_norm + ttnn elementwise
  |-- [QKV matmul]                           ttnn.matmul (72-core parallel)
  |-- [RoPE + layout transform]              TT-Lang (rope_layout_kernel)
  |-- [SDPA]                                 ttnn.scaled_dot_product_attention
  |-- [O projection]                         ttnn.matmul (+ all_reduce for multi-chip)
  |-- [gated residual]                       TT-Lang (gated_residual_kernel)
  |-- [LayerNorm + adaLN modulate]           ttnn.layer_norm + ttnn elementwise
  |-- [FC1 + GELU]                           ttnn.linear (fused activation)
  |-- [FC2]                                  ttnn.matmul (+ all_reduce for multi-chip)
  |-- [gated residual]                       TT-Lang (gated_residual_kernel)

Final layer:
  |-- [adaLN params]                         ttnn.linear
  |-- [LayerNorm]                            TT-Lang (layernorm_d1024)
  |-- [adaLN modulate]                       TT-Lang (adaln_modulate_kernel)
  |-- [output projection]                    TT-Lang (linear_k32)
```

### VAE decode (per frame)

The VAE decoder is a 12-block ViT with 2D pixel-based RoPE (not learned axial like the DiT). It runs entirely on device inside the same trace as the DDIM loop.

```
Latent (576, 32)
  |-- [post_quant_conv]                      ttnn.linear
  |
  x12 decoder blocks:
  |-- [LayerNorm]                            TT-Lang (layernorm_d1024)
  |-- [QKV + bias]                           ttnn.linear
  |-- [RoPE + QKV split + layout]            TT-Lang (vae_rope_fused)
  |-- [SDPA]                                 ttnn.scaled_dot_product_attention
  |-- [O projection + bias]                  ttnn.linear
  |-- [residual add]                         ttnn.add
  |-- [LayerNorm]                            TT-Lang (layernorm_d1024)
  |-- [FC1 + GELU]                           ttnn.linear (fused activation)
  |-- [FC2 + bias]                           ttnn.linear
  |-- [residual add]                         ttnn.add
  |
  |-- [final LayerNorm]                      TT-Lang (layernorm_d1024)
  |-- [predictor linear]                     ttnn.linear
  |
  v
Pixel patches (576, 1200)
```

### TT-Lang kernel summary

The pipeline uses a hybrid strategy: **ttnn** for large matmuls and sdpa (for performance and precision) and **TT-Lang** for fused elementwise ops, custom data layout transforms, and operations where fusion eliminates DRAM round-trips.

| Kernel | File | What it does | Where used |
|--------|------|-------------|------------|
| `silu_kernel` | `src/silu.py` | `x * sigmoid(x)` | Conditioning (1x per step) |
| `gated_residual_kernel` | `src/gated_residual.py` | `residual + x * gate` | DiT sub-block residuals (32x per step) |
| `adaln_modulate_kernel` | `src/adaln_modulate.py` | `x * (1 + scale) + shift` | DiT final layer |
| `layernorm_d1024` | `src/layernorm.py` | 3-pass LN with affine | DiT final layer + VAE (26x per step) |
| `linear_k32` | `src/linear.py` | Matmul with K-accumulation | DiT final output projection |
| `rope_layout_spatial` | `src/rope_layout_kernel.py` | Fused RoPE + QKV split + SDPA layout | DiT spatial attention (16x per step) |
| `rope_temporal` | `src/rope_layout_kernel.py` | Fused temporal RoPE + QKV split | DiT temporal attention (16x per step) |
| `vae_rope_fused` | `src/vae_rope.py` | Fused 2D RoPE + QKV split + layout | VAE attention (12x per step) |
| `adaln_matmul_expand_kernel` | `src/adaln_matmul_expand.py` | adaLN matmul + bias + row expand | Currently unused (ttnn.linear used instead) |

### The rope_layout_kernel: TT-Lang data movement in action

The `rope_layout_kernel` is the best example of where TT-Lang's explicit data movement control outperforms a ttnn op sequence. After the QKV matmul, the combined tensor is `(SEQ, 5*D_MODEL)` with columns laid out as `[Q | K | V | Q_swap | K_swap]`. To feed ttnn SDPA, we need Q, K, V each in `(BATCH*HEADS, 1, SEQ, D_HEAD)` format with RoPE applied to Q and K.

Without TT-Lang, this requires 5 `ttnn.slice` ops to extract Q/K/V/Q_swap/K_swap, 2 fused RoPE kernels (multiply + add), and 9 reshape/permute ops to get the SDPA layout -- 16 ops total, each with its own DRAM round-trip.

The TT-Lang kernel does this in a single pass: the **data movement thread** reads the right columns from the packed QKV tensor (using computed offsets per head), and the **compute thread** applies RoPE in-flight. The write-back goes directly to the SDPA-format output tensors. This eliminates ~15 intermediate DRAM transfers per attention head per sub-block, which adds up across 32 sub-blocks per DDIM step.

### In-progress kernels (not currently used)

Several TT-Lang kernels were written that fuse larger portions of the pipeline and leverage pipes but are currently superseded by ttnn equivalents that achieve better core utilization or accuracy. These are preserved in `src/in_progress.py` for future optimization:

| Kernel | What it fuses | Why unused |
|--------|--------------|------------|
| `make_fused_ln_adaln_kernel` | LayerNorm + adaLN modulate | 2x slower than ttnn.layer_norm (0.3ms vs 0.15ms) |
| `make_fused_gated_res_ln_adaln_kernel` | Gated residual + LN + adaLN | 5.5x slower than ttnn (0.8ms vs 0.15ms) |
| `make_qkv_rope_sdpa_kernel` | QKV matmul + RoPE + SDPA | K-accumulation matmul only uses 32 cores; ttnn.matmul uses 72 |
| `make_temporal_qkv_rope_sdpa_kernel` | Temporal QKV + RoPE + SDPA | Same core utilization issue |
| `make_mega_post_attn_kernel` | O proj + LN + FC1 + GELU + FC2 + residual | Per-row processing only uses 10 cores |
| `make_linear_bias_kernel` | Matmul + bias add | Replaced by ttnn.linear |
| `make_fused_linear_bias_gelu_kernel` | Matmul + bias + GELU | Replaced by ttnn.linear with fused activation |
| `make_fused_linear_bias_gated_res_kernel` | Matmul + bias + gate + residual | Replaced by ttnn.matmul + TT-Lang gated_residual |
| `make_rope_sdpa_kernel` | RoPE + SDPA (no QKV matmul) | Replaced by rope_layout + ttnn.SDPA |

### What runs on host

- **VAE encode** -- full encoder forward pass for the prompt image (runs once at startup)
- **Patch embed** -- Conv2d embedding of latents into patch tokens (runs once per DDIM step via matmul on device, but the initial prompt uses CPU conv2d)
- **Conditioning** -- timestep + action embedding MLP (fp32 on host for numerical precision, ~0.1ms)
- **Unpatchify** -- reshape VAE output patches into image pixels (CPU, negligible)
- **Context window management** -- sliding window bookkeeping between frames (CPU)

  linear.py                 # Matmul kernel (compiler K-accumulation)

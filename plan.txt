MODEL ARCHITECTURE
------------------
Input: 360x640 RGB -> VAE encode -> 18x32x16 latent
DiT: 16 SpatioTemporalDiTBlocks
  Each block has:
    Spatial sub-block:
      - LayerNorm(1024, eps=1e-6) + adaLN (shift, scale, gate)
      - SpatialAxialAttention: QKV(no bias, 1024->3072), RoPE(pixel), SDPA
      - O projection: Linear(1024, 1024)
      - MLP: Linear(1024, 4096) -> GELU(tanh) -> Linear(4096, 1024)
      - adaLN (shift, scale, gate) for MLP too
    Temporal sub-block: same structure but causal attention
  Conditioning: TimestepEmbedder(sinusoidal 256 -> MLP 1024) + external_cond(25->1024)
  FinalLayer: LayerNorm + adaLN(2 params) + Linear(1024, 64)
Output: unpatchify -> VAE decode -> 360x640 RGB

Diffusion: DDIM with sigmoid beta schedule, 10 steps default

KEY DIMENSIONS

DESIGN DECISIONS
----------------
1. VAE on CPU (PyTorch) for now - it runs once per frame, complex conv2d.
   The DiT is the hot path (10 DDIM steps per frame).

2. DiT on TT device via TT-Lang kernels + ttnn ops for reshaping/SDPA.

3. Follow toy-wm patterns closely:
   - grid="auto" + streaming loops for all kernels
   - Weight preloading to DRAM at startup
   - Scratch buffer pre-allocation and reuse
   - ttnn.transformer.scaled_dot_product_attention for SDPA
   - Host-side for reshape/permute between attention formats

4. Oasis uses LayerNorm (not RMSNorm) - needs mean subtraction + bias.
   3-pass kernel: mean, variance, normalize.

5. Oasis uses GELU(approximate="tanh") not GEGLU/SiLU for MLP activation.

6. Oasis QKV has NO bias (unlike toy-wm). Separate spatial/temporal QKV.

7. For MLP down (4096->1024), K=128 tiles. Use accumulating linear kernel
   with k_chunk=32, k_iters=4.

KERNEL INVENTORY
================
  ELEM_GRAN = 8  (divides 32 for d=1024 and 128 for d=4096)

  Linear kernels (from toy-wm pattern):
    make_linear_kernel(k_chunk)          - single-pass matmul, no bias
    make_linear_bias_kernel(k_chunk, n_chunk) - matmul + bias
    make_linear_accum_kernel(k_chunk, k_iters) - multi-pass K accumulation

  Elementwise kernels:
    silu_kernel           - x * sigmoid(x), for timestep embedder
    gelu_approx_kernel    - tanh-approximate GELU for MLP
    adaln_modulate_kernel - x * (1 + scale) + shift
    gated_residual_kernel - residual + gate * x

  Norm kernel:
    make_layernorm_kernel(dim_tiles) - 3-pass LayerNorm with weight+bias

  Fused kernels (later):
    make_fused_ln_modulate_kernel(dim_tiles) - LayerNorm + adaLN in one
    make_linear_bias_gated_res_kernel(k_chunk) - matmul + bias + gated residual
    mod_broadcast_kernel - broadcast 6/12 adaLN params from conditioning

DEVELOPMENT WORKFLOW
--------------------
1. Write Python scripts locally in /Users/zcarver/Developer/oasis-ttlang/
2. Run them on the remote via: ~/.claude/commands/tools/run-test.sh --hw /path/to/script.py
   - run-test.sh copies the file to /tmp/ttl_test_temp.py on the remote and executes it
   - Output goes to /tmp/ttlang_test_output.log on the remote
   - Read logs via: ~/.claude/commands/tools/remote-run.sh cat /tmp/ttlang_test_output.log
   - Use grep/tail locally: remote-run.sh cat /tmp/ttlang_test_output.log | grep "PASS\|FAIL"
3. For utility scripts (e.g., inspect_weights.py that don't need ttl):
   - Still use run-test.sh --hw to run them. It works for any Python script.
   - The script runs in /tmp on the remote, so all file paths must be absolute remote paths.
4. If a script needs dependencies (other .py files), use copy-file.sh to copy them
   to /tmp first. Everything runs out of /tmp on the remote.
5. DO NOT use remote-run.sh with inline Python (-c). Quoting through the SSH+docker
   chain is broken for anything non-trivial. Always write a .py file and use run-test.sh.
6. The simulator (default, no --hw) is faster for iteration but has known false positives
   for buffer_factor=1 DFBs held across loops. Use --hw for those kernels.
7. Exit code 0 does NOT mean success - always read the log for PASS/FAIL/errors.

KNOWN ISSUES / GOTCHAS
-----------------------
1. SIM "pop() after WAIT" ERROR: buffer_factor=1 DFBs held in `with` scope
   across a loop trigger a false positive in the simulator. The pattern works
   on HW. When you see "Cannot perform pop(): Expected STORE_SRC", skip sim
   and use --hw flag. This affects LayerNorm and any kernel that holds
   constants (scaler, mean_scale) across the main loop.

2. ALL DEFINED DFBs MUST BE USED. If you define a DFB but never use it,
   you get "Missing CB config for index N". Every DFB needs reserve/wait.

3. NO STRING CAPTURES IN KERNELS. Factory functions cannot capture strings.
   Use numeric constants only.

4. ttnn SDPA REQUIRES TILE-ALIGNED d_head. d_head=64 is already 2 tiles,
   so no padding needed for Oasis (unlike toy-wm's d_head=16).

5. ttnn SDPA SCALE PARAMETER. If the model uses any form of QK normalization,
   must explicitly pass scale=1.0. Oasis does NOT use QK-norm, so the default
   scale (1/sqrt(d_head)) should be correct. Verify empirically.

6. COPY_DST LIMITATION. Expressions reusing intermediates can fail:
   e.g., s = a + b; result = s * sigmoid(s) may not compile.
   Workaround: split into multiple with blocks with intermediate DFBs.

7. SPATIAL vs TEMPORAL ATTENTION RESHAPE. The 5D tensor (B,T,H,W,D) needs
   different reshaping for spatial (B*T, heads, H*W, d) vs temporal
   (B*H*W, heads, T, d). Use ttnn reshape/permute for this.

8. ROTARY EMBEDDINGS. Oasis uses two types:
   - Spatial: pixel-based freqs (RotaryEmbedding(dim=32, freqs_for="pixel"))
   - Temporal: standard lang freqs (RotaryEmbedding(dim=64))
   Precompute on host, send to device as tensors.

  Total: ~1.5-2.5s per frame

T=32 sliding window:
  DiT forward: ~4.1 TFLOP matmul -> ~200-400ms/step
  10 DDIM steps: ~2-4s per frame

8 chips with tensor parallelism:
  Target: 4-8 fps across sliding window
